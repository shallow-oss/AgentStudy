# AI 应用开发工程师岗位学习路线与实战手册

> 目标：围绕 AI 应用开发工程师岗位要求，系统补齐后端工程、LLM 应用、RAG、Function Calling、Workflow、Agent、工程化落地与项目表达能力，最终形成可展示、可面试、可写进简历的作品集。

## 1. 岗位能力拆解

这个岗位不是纯算法研究岗，也不是只会调用大模型 API 的工具使用岗。它更接近：

- AI 应用开发工程师
- LLM 应用后端工程师
- RAG / Agent 应用工程师
- AI 产品落地工程师

岗位核心要求可以概括为一句话：

> 能把大模型能力接入真实业务场景，并通过工程手段做成可用、稳定、可评估、可持续迭代的产品功能。

### 1.1 必备能力

| 能力方向 | 具体要求 | 对应岗位描述 |
| --- | --- | --- |
| 后端开发 | Python / Java / Go、API、数据库、缓存、日志、服务调试 | 模型接入、后端服务开发、接口设计 |
| LLM 基础应用 | Prompt、多轮对话、流式输出、JSON 输出、工具调用 | 智能问答、内容生成、对话交互 |
| RAG | 文档解析、切分、Embedding、向量数据库、召回、重排、引用来源 | 知识检索、企业知识库问答 |
| Function Calling | 让模型调用业务接口、数据库、外部工具 | 智能体、自动化流程、业务操作 |
| Workflow / Agent | 多步骤任务编排、状态管理、工具选择、失败重试 | 流程自动化、复杂任务执行 |
| 工程化 | 日志监控、效果评估、成本控制、稳定性优化 | 可用、好用、持续演进 |
| 产品意识 | 业务理解、用户体验、需求拆解、方案表达 | 与产品、算法、设计、业务团队协作 |

### 1.2 学习优先级

优先学习：

1. Python 后端开发
2. 大模型 API 调用
3. Prompt Engineering
4. RAG 知识库问答
5. Function Calling / Tool Calling
6. Workflow / Agent
7. 日志、评估、稳定性和成本优化
8. 项目包装与面试表达

暂时不必优先深挖：

- 大模型预训练
- Transformer 数学推导
- 分布式训练
- 模型微调底层原理
- CUDA / GPU 训练优化

这些内容有价值，但不是该岗位短期入门和求职的主线。

## 2. 总体学习规划

建议周期：约 10-14 周。

如果每天学习 2-3 小时，可以按 3 个月规划；如果时间更充足，可以压缩到 6-8 周。

| 阶段 | 时间 | 核心目标 | 主要产出 |
| --- | --- | --- | --- |
| 阶段 1 | 2-3 周 | 补齐后端工程基础 | 一个完整后端 API 项目 |
| 阶段 2 | 1-2 周 | 掌握 LLM API 和 Prompt | 一个 AI 对话助手 |
| 阶段 3 | 3-4 周 | 做出 RAG 知识库问答系统 | 一个核心作品集项目 |
| 阶段 4 | 3-4 周 | 掌握 Workflow / Agent | 一个业务自动化 AI 应用 |
| 阶段 5 | 2-3 周 | 工程化、优化和作品集包装 | 可部署 Demo、README、面试材料 |

## 3. 阶段 1：后端工程基础

### 3.1 阶段目标

你需要先具备独立开发后端服务的能力。AI 应用最终还是要落到接口、数据库、任务、日志、部署和稳定性上。

完成本阶段后，你应该能够：

- 使用 Python 编写后端 API。
- 设计基本的 RESTful 接口。
- 使用数据库保存业务数据。
- 使用 Redis 做缓存或临时状态存储。
- 使用 Docker 启动服务。
- 编写清晰的项目结构。
- 通过日志定位问题。
- 写出基础 README 和接口文档。

### 3.2 推荐技术栈

优先建议使用 Python，因为 AI 应用生态更完整。

| 模块 | 推荐工具 |
| --- | --- |
| Web 框架 | FastAPI |
| 数据库 | PostgreSQL 或 MySQL |
| ORM | SQLAlchemy |
| 缓存 | Redis |
| 配置管理 | python-dotenv / pydantic-settings |
| 接口文档 | FastAPI Swagger |
| 依赖管理 | uv / pip / poetry 任选一个 |
| 部署 | Docker / Docker Compose |
| 测试 | pytest |

## 4. 阶段 2：LLM API 与 Prompt

### 4.1 阶段目标

本阶段要从普通后端服务升级为 AI 后端服务。

完成本阶段后，你应该能够：

- 调用大模型 API。
- 构造 System Prompt 和 User Prompt。
- 处理多轮对话上下文。
- 实现流式输出。
- 让模型返回结构化 JSON。
- 使用 Function Calling 调用外部工具。
- 保存模型输入、输出、耗时和 token 消耗。

### 4.2 学习清单

#### 大模型 API 基础

需要掌握：

- API Key 配置
- Chat Completion
- System / User / Assistant 消息结构
- temperature、top_p、max_tokens
- 流式输出
- 超时处理
- 重试机制

验收标准：

- 能写一个后端接口 `/api/chat`，接收用户问题并返回模型回答。
- 能实现流式返回。
- API Key 不写死在代码里，而是放在环境变量中。

#### Prompt Engineering

需要掌握：

- 角色设定
- 任务描述
- 输出格式约束
- 示例引导
- 边界条件
- 反幻觉约束
- 多轮上下文压缩

常用 Prompt 结构：

```text
你是一个{角色}。

你的任务是：
1. {任务 1}
2. {任务 2}

请遵守以下规则：
1. {规则 1}
2. {规则 2}

请按照以下 JSON 格式输出：
{
  "summary": "...",
  "items": []
}
```

验收标准：

- 同一个问题在 Prompt 优化前后，回答质量有明显差别。
- 能让模型稳定输出指定 JSON 结构。
- 能解释 Prompt 中每一部分的作用。

#### 多轮对话

需要掌握：

- 对话历史存储
- 上下文窗口限制
- 历史消息裁剪
- 摘要式记忆
- 会话 ID 管理

验收标准：

- 用户可以连续追问。
- 系统能根据前文回答。
- 对话历史可以保存到数据库。

#### Function Calling / Tool Calling

需要掌握：

- 定义工具名称
- 定义工具参数 schema
- 判断模型是否要调用工具
- 执行工具函数
- 把工具结果返回给模型生成最终答案

示例工具：

- 查询订单
- 查询天气
- 查询数据库
- 发送邮件
- 生成报表
- 搜索知识库

验收标准：

- 用户问“帮我查一下订单 1001 的状态”，模型能调用查询订单工具。
- 工具返回结果后，模型能组织成自然语言回答。
- 工具调用过程有日志记录。

## 5. 阶段 3：RAG 知识库问答

### 5.1 阶段目标

RAG 是这个岗位最核心的能力之一。很多企业 AI 应用都不是让模型凭空回答，而是让模型基于企业内部文档、制度、FAQ、产品手册、客服知识库进行回答。

完成本阶段后，你应该能够：

- 解析 PDF、Word、Markdown、TXT 文档。
- 对文档进行切分。
- 生成 Embedding。
- 存入向量数据库。
- 根据用户问题召回相关片段。
- 让模型基于检索结果回答。
- 在回答中展示引用来源。
- 评估和优化召回质量。

### 5.2 RAG 基本流程

```text
文档上传
  -> 文档解析
  -> 文本清洗
  -> 文本切分
  -> 生成 Embedding
  -> 写入向量数据库
  -> 用户提问
  -> 问题向量化
  -> 相似度检索
  -> 可选重排
  -> 构造 Prompt
  -> 大模型生成答案
  -> 返回答案和引用来源
```

### 5.3 学习清单

#### 文档解析

需要掌握：

- TXT / Markdown 解析
- PDF 解析
- Word 解析
- 表格处理
- 页码和标题保留
- 文档元数据保存

验收标准：

- 上传一份 PDF 后，可以抽取正文内容。
- 抽取结果能保留文档名称、页码、段落位置等元数据。

#### 文本切分

需要掌握：

- 按固定长度切分
- 按标题切分
- 按段落切分
- chunk size
- overlap
- 语义完整性

常见配置：

```text
chunk_size: 500-1000 中文字符
chunk_overlap: 50-150 中文字符
```

切分时要注意：

- 不要把一句话切断。
- 不要把标题和正文分离得太远。
- 不要让 chunk 太短，否则上下文不足。
- 不要让 chunk 太长，否则召回不精准。

验收标准：

- 能查看每个 chunk 的内容。
- 能解释为什么这样设置 chunk size 和 overlap。

#### Embedding

需要掌握：

- Embedding 的作用
- 查询向量和文档向量
- 相似度计算
- 中文语义检索
- Embedding 模型选择

验收标准：

- 能把文本转成向量。
- 能用相似度找到相关文本片段。

#### 向量数据库

可选工具：

- Chroma：适合入门、本地 Demo。
- FAISS：适合本地高性能向量检索。
- Milvus：适合更正式的向量数据库场景。
- Qdrant：API 友好，工程化体验较好。

初学建议：

> 先用 Chroma 或 FAISS，后续再了解 Milvus / Qdrant。

需要掌握：

- collection
- document
- metadata
- top_k
- similarity score
- insert / search / delete / update

验收标准：

- 能把文档 chunk 写入向量库。
- 能通过用户问题检索 top_k 相关片段。

#### 检索增强生成

需要掌握：

- 将检索片段插入 Prompt。
- 限制模型只能根据上下文回答。
- 回答中附带引用来源。
- 当检索不到相关内容时，要求模型承认不知道。

推荐 Prompt 结构：

```text
你是企业知识库问答助手。

请只根据【知识库内容】回答用户问题。
如果知识库内容不足以回答，请明确说明“根据当前知识库无法确定”。
不要编造不存在的信息。

【知识库内容】
{retrieved_context}

【用户问题】
{question}

请输出：
1. 答案
2. 引用来源
```

验收标准：

- 回答必须基于召回内容。
- 回答能附带来源。
- 检索不到时不会强行编造。

#### RAG 评估

需要关注：

- 是否召回到了正确文档。
- 是否召回到了正确段落。
- 回答是否忠实于原文。
- 是否出现幻觉。
- 响应时间是否可接受。
- 每次问答成本是多少。

建议建立测试集：

| 问题 | 标准答案 | 应命中文档 | 应命中段落 |
| --- | --- | --- | --- |
| 公司年假规则是什么？ | 根据制度文件回答 | 员工手册.pdf | 第 3 节 |
| 报销需要哪些材料？ | 根据财务制度回答 | 报销制度.pdf | 第 2 节 |

验收标准：

- 至少准备 20 个测试问题。
- 记录每个问题是否召回正确。
- 对比不同 chunk_size、top_k、Prompt 的效果。

### 5.4 阶段项目：企业知识库问答系统

项目名称：

> KnowledgeQA：企业知识库 RAG 问答系统

核心功能：

- 文档上传
- 文档解析
- 文档切分
- 向量化入库
- 知识库检索
- 基于知识库问答
- 答案引用来源
- 检索日志
- 模型调用日志
- 测试问题集评估

建议接口：

| 方法 | 路径 | 功能 |
| --- | --- | --- |
| POST | /api/documents/upload | 上传文档 |
| GET | /api/documents | 文档列表 |
| GET | /api/documents/{id}/chunks | 查看文档切片 |
| POST | /api/knowledge/query | 知识库问答 |
| POST | /api/knowledge/search | 只检索不生成 |
| GET | /api/rag/logs | RAG 调用日志 |
| POST | /api/evaluation/run | 执行测试集评估 |

建议数据表：

- documents：文档信息
- document_chunks：文档切片
- conversations：问答会话
- messages：消息记录
- rag_logs：检索日志
- llm_logs：模型调用日志
- evaluation_cases：评估问题集
- evaluation_results：评估结果

### 5.5 进阶优化

基础版完成后，可以逐步加入：

- 混合检索：关键词检索 + 向量检索。
- Rerank：对召回结果重新排序。
- Query Rewrite：改写用户问题，提高召回率。
- Multi-query：生成多个相似问题进行检索。
- Parent-child chunk：小块检索，大块生成。
- 文档权限控制：不同用户只能查自己的知识库。
- 答案置信度提示。
- 引用高亮。

### 5.6 阶段验收标准

完成后检查：

- 支持上传至少一种文档格式。
- 文档能被切分并写入向量库。
- 用户提问能召回相关片段。
- 模型回答带引用来源。
- 检索和模型调用有日志。
- 有至少 20 个测试问题。
- 能展示一次优化前后效果对比。

## 6. 阶段 4：Workflow 与 Agent

### 6.1 阶段目标

RAG 解决的是“基于知识回答问题”，Workflow 和 Agent 解决的是“让 AI 按步骤完成任务”。

完成本阶段后，你应该能够：

- 把复杂任务拆成多个节点。
- 设计 AI 工作流。
- 让模型调用多个工具。
- 管理任务状态。
- 加入人工确认节点。
- 处理失败、重试和兜底。
- 使用 LangGraph、Dify、Coze 等工具完成 AI 应用编排。

### 6.2 Workflow 和 Agent 的区别

| 类型 | 特点 | 适用场景 |
| --- | --- | --- |
| Workflow | 流程固定，节点明确，可控性强 | 审批、报告生成、客服流程、内容生产 |
| Agent | 模型自主规划步骤和选择工具，灵活性强 | 开放任务、复杂推理、多工具协作 |

初学建议：

> 先学 Workflow，再学 Agent。

原因：

- Workflow 更稳定。
- 更容易解释。
- 更适合企业业务落地。
- 面试时更容易讲清楚方案。

### 6.3 学习清单

#### 工作流拆解

需要掌握：

- 输入节点
- 信息抽取节点
- 条件判断节点
- 工具调用节点
- 模型生成节点
- 人工确认节点
- 输出节点

示例：

```text
上传简历和 JD
  -> 提取 JD 要求
  -> 提取简历信息
  -> 能力匹配评分
  -> 生成面试问题
  -> 人工确认
  -> 生成评估报告
```

验收标准：

- 能把一个业务需求拆成 5-8 个清晰节点。
- 能说明每个节点输入、输出和失败处理。

#### LangGraph / Dify

建议二选一或都了解：

- LangGraph：更偏代码，适合后端工程师。
- Dify：更偏平台化和低代码，适合快速 Demo。

需要掌握：

- 节点
- 边
- 条件分支
- 状态
- 工具调用
- 工作流日志
- 测试和调试

验收标准：

- 能搭建一个多节点 AI 工作流。
- 每个节点的输出可以被后续节点使用。
- 失败时能定位是哪个节点的问题。

#### Agent

需要掌握：

- 工具列表
- 任务规划
- 工具选择
- 工具调用结果观察
- 多步执行
- 停止条件
- 最大执行轮数

注意事项：

- Agent 不一定比 Workflow 更好。
- Agent 的不确定性更高。
- 生产业务中通常需要约束 Agent 的工具、权限和执行步数。

验收标准：

- 能让 Agent 在限定工具内完成一个任务。
- 能限制最大调用次数。
- 能记录每一步执行过程。

- 设计订单查询工具 schema，使模型能够按需调用业务接口。
- 保存对话历史、工具调用记录和模型调用日志。
- 处理模型超时、工具失败和输出格式异常等问题。

- 是否能 3 分钟讲清楚架构？

## 13. 最终达标标准

- 有 2 个可以展示的 AI 应用项目。
- 能讲清楚项目的业务价值、技术方案、踩坑和优化。

## 14. 建议的学习心态

这个岗位最看重的是动手和落地。

不要把学习路线变成只看课、只收藏资料、只了解概念。每学一个知识点，都要问自己：

- 我能不能用代码跑起来？
- 我能不能接到一个真实业务场景？
- 我能不能解释为什么这样设计？
- 我能不能记录效果并做优化？
- 我能不能把它写进简历并在面试中讲清楚？

最重要的目标不是“学完所有技术”，而是：

> 做出 2 个真正能跑、能展示、能解释、能优化的 AI 应用项目。
